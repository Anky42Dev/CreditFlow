## 1. Общая информация

### 1.1 Описание проекта

**CreditFlow** — цифровая платформа потребительского кредитования (consumer lending platform), позволяющая физическим лицам полностью онлайн подобрать кредитный продукт, подать заявку, пройти автоматический скоринг, получить одобрение и обслуживать кредит (график платежей, погашения, история операций) без визита в отделение.

Система реализует полный жизненный цикл кредита:

```
Регистрация клиента
   → Верификация профиля
   → Выбор кредитного продукта
   → Подача кредитной заявки
   → Автоматический скоринг (риск-оценка)
   → Решение (одобрено / отказано / на ручной проверке)
   → Выдача кредита (создание кредитного договора)
   → Формирование графика платежей
   → Обслуживание (погашения, транзакции, уведомления)
   → Закрытие кредита
```

CreditFlow построен так, чтобы **один и тот же кодовый домен** постепенно наращивался: Junior реализует ядро (регистрация + заявки + продукты), Middle добавляет обработку, роли и real-time, Senior превращает систему в production-ready с очередями, скоринг-воркерами, мониторингом и масштабированием.

### 1.2 Бизнес-задача и цели

**Бизнес-задача.** Классическая выдача потребительских кредитов через отделения — медленная, дорогая и не масштабируется. Банку нужен цифровой канал, который:

- снижает стоимость обработки одной заявки (cost per application);
- сокращает time-to-decision с дней до секунд за счёт автоматического скоринга;
- увеличивает объём выдач без роста штата операторов;
- обеспечивает прозрачность и аудит каждого решения (регуляторное требование).

**Бизнес-цели (измеримые):**

| Цель | Метрика | Целевое значение |
|---|---|---|
| Ускорить принятие решения | Median time-to-decision | < 30 секунд для автоскоринга |
| Снизить нагрузку на операторов | Доля заявок без ручной обработки | ≥ 70% |
| Обеспечить доступность | Uptime основного API | 99.9% |
| Прозрачность решений | Покрытие Audit Log | 100% операций изменения статуса |
| Конверсия | Заявка → выдача | ≥ 40% от одобренных |

**Продуктовые цели:**

- Единый личный кабинет клиента: заявки, договоры, график платежей, история.
- Рабочее место оператора/андеррайтера: очередь заявок на ручную проверку.
- Административная панель: управление продуктами, ролями, аудит.

### 1.3 Целевая аудитория

**Конечные потребители (B2C):**
- Физические лица 18–65 лет, желающие оформить потребительский кредит онлайн.

**Внутренние пользователи (сотрудники банка):**
- Операторы поддержки (Support) — консультируют, видят статусы.
- Андеррайтеры / кредитные специалисты — обрабатывают заявки на ручной проверке.
- Администраторы — управляют продуктами, пользователями, системными настройками.

### 1.4 Роли пользователей

Единая ролевая модель, сквозная для всех уровней (на Junior используется упрощённое подмножество, на Middle вводится полноценный RBAC).

| Роль | Код | Появляется на уровне | Описание |
|---|---|---|---|
| Гость | `GUEST` | Junior | Неавторизованный. Видит публичные продукты, может регистрироваться. |
| Клиент | `CLIENT` | Junior | Зарегистрированный. Управляет профилем, подаёт заявки, обслуживает кредиты. |
| Оператор поддержки | `SUPPORT` | Middle | Просмотр заявок и клиентов (read-mostly). |
| Андеррайтер | `UNDERWRITER` | Middle | Ручная обработка заявок: одобрение/отказ, запрос документов. |
| Администратор | `ADMIN` | Middle | Полный доступ: продукты, роли, пользователи, аудит. |

> На **Junior** реально существуют только `GUEST` и `CLIENT`; поле роли уже присутствует в модели `User` (заложено на будущее), но RBAC-проверок ещё нет — есть только owner-based access («свой/чужой»). Схема БД не меняется между уровнями — меняется только логика авторизации.

### 1.5 Глоссарий предметной области

| Термин | Определение |
|---|---|
| **Кредитный продукт** (Credit Product) | Шаблон условий кредита: сумма (min/max), ставка, срок, требования. Настраивается администратором. |
| **Кредитная заявка** (Credit Application) | Запрос клиента на получение кредита по конкретному продукту с указанием суммы и срока. |
| **Скоринг** (Scoring) | Автоматическая оценка кредитоспособности; возвращает балл и рекомендацию. |
| **Андеррайтинг** (Underwriting) | Процесс принятия решения по заявке (авто или ручной). |
| **Кредитный договор** (Loan / Contract) | Созданный после одобрения объект: тело долга, ставка, срок, статус. |
| **График платежей** (Payment Schedule) | Список плановых аннуитетных платежей с датами и суммами. |
| **Аннуитетный платёж** | Равный ежемесячный платёж (часть тела долга + проценты). |
| **Транзакция** (Transaction) | Финансовая операция: выдача, погашение, начисление процентов. |
| **Статус заявки** | Машина состояний: `DRAFT → SUBMITTED → SCORING → APPROVED / REJECTED / MANUAL_REVIEW → DISBURSED`. |
| **Audit Log** | Неизменяемый журнал значимых действий (кто, что, когда, до/после). |

## 2. Product Vision

### 2.1 Основные бизнес-сценарии (User Journeys)

**UJ-1. Клиент оформляет кредит (happy path):**
```
Гость открывает каталог продуктов
  → Регистрируется (email + пароль)
  → Логинится, получает JWT
  → Заполняет профиль (ФИО, дата рождения, доход)
  → Выбирает продукт "Потребительский кредит"
  → Создаёт заявку: сумма 200 000, срок 12 мес
  → Отправляет заявку (SUBMITTED)
  → Система запускает скоринг (SCORING)
  → Скоринг возвращает APPROVED
  → Создаётся кредитный договор + график платежей
  → Клиент видит график и первый платёж
```

**UJ-2. Заявка уходит на ручную проверку (Middle+):**
```
Клиент подаёт заявку на крупную сумму
  → Скоринг возвращает MANUAL_REVIEW
  → Заявка попадает в очередь андеррайтера
  → Андеррайтер запрашивает документы
  → Клиент загружает справку о доходах
  → Андеррайтер одобряет → DISBURSED
  → Клиент получает WebSocket-уведомление в реальном времени
```

**UJ-3. Клиент обслуживает кредит:**
```
Клиент открывает активный договор
  → Видит график платежей и ближайший платёж
  → Совершает погашение
  → Создаётся транзакция REPAYMENT
  → Обновляется остаток долга и статус платежа
  → При полном погашении договор → CLOSED
```

**UJ-4. Администратор управляет каталогом (Middle+):**
```
Админ создаёт новый кредитный продукт
  → Задаёт сумму, ставку, срок, требования
  → Публикует продукт (is_active = true)
  → Продукт становится доступен клиентам
  → Все действия фиксируются в Audit Log
```

### 2.2 Границы системы (Scope)

**Входит в систему:**
- Регистрация, аутентификация, профиль клиента.
- Каталог кредитных продуктов.
- Подача и обработка кредитных заявок.
- Скоринг (эмулированный/rule-based, не реальное БКИ).
- Кредитные договоры и графики платежей.
- Транзакции и погашения.
- Уведомления, админ-панель, аудит.

**НЕ входит (Out of scope):**
- Реальная интеграция с банковской core-системой и процессингом карт.
- Реальный запрос в бюро кредитных историй (эмулируется).
- KYC/AML-проверки с гос. сервисами (эмулируется полем статуса).
- Бухгалтерский учёт и налоговая отчётность.
- Мобильные нативные приложения (только веб).

### 2.3 Дорожная карта эволюции уровней

```
JUNIOR (Ядро)
  ├── Auth: регистрация, логин, JWT + refresh
  ├── Профиль клиента + загрузка аватара
  ├── Каталог продуктов (чтение)
  ├── Заявки: CRUD (DRAFT → SUBMITTED)
  ├── Простой синхронный псевдо-скоринг
  └── Pagination / Filter / Sort / Search

MIDDLE (Обработка + Real-time)  [расширяет ту же схему]
  ├── RBAC: роли SUPPORT / UNDERWRITER / ADMIN
  ├── Админ-панель (управление продуктами, пользователями)
  ├── Асинхронный скоринг через Celery + Redis
  ├── WebSocket (Django Channels): статусы заявок, уведомления
  ├── Кредитные договоры + графики платежей + транзакции
  ├── Кэширование (Redis), сложная фильтрация, оптимизация запросов
  ├── Email-уведомления, Cron-задачи
  ├── Docker + docker-compose + CI/CD
  └── Audit Log

SENIOR (Production-ready)  [рефакторинг в Clean Architecture]
  ├── Clean Architecture + DDD (Aggregates, Domain Events)
  ├── Repository Pattern, Service Layer, DI
  ├── Выделение скоринг-сервиса в FastAPI (async)
  ├── RabbitMQ/Kafka для доменных событий, Saga
  ├── S3 для документов, обработка изображений
  ├── Prometheus (метрики), OpenTelemetry (трассировка)
  ├── OWASP, Rate Limit, Feature Flags, Health Checks
  ├── Оптимизация БД (индексы, партиционирование)
  └── Docker Compose (prod) + Kubernetes (концептуально)
```

## 3. Единая доменная модель (сквозная)

### 3.1 Bounded Contexts

```
┌─────────────────────────────────────────────────────────────┐
│                        CreditFlow                            │
│                                                              │
│  ┌──────────────┐   ┌──────────────┐   ┌─────────────────┐   │
│  │   IDENTITY   │   │   LENDING    │   │    PAYMENTS     │   │
│  │  (Auth)      │   │  (Credit)    │   │                 │   │
│  │              │   │              │   │                 │   │
│  │ User         │   │ CreditProduct│   │ Loan            │   │
│  │ Profile      │   │ Application  │   │ PaymentSchedule │   │
│  │ Role         │   │ ScoringResult│   │ Transaction     │   │
│  │ Permission   │   │ Document     │   │                 │   │
│  └──────────────┘   └──────────────┘   └─────────────────┘   │
│                                                              │
│  ┌──────────────┐   ┌──────────────────────────────────┐    │
│  │ NOTIFICATION │   │           PLATFORM               │    │
│  │              │   │  (сквозные технические сущности) │    │
│  │ Notification │   │  AuditLog, FeatureFlag           │    │
│  └──────────────┘   └──────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Master ER-диаграмма (текстовая)

Обозначения: `1--*` = один-ко-многим, `1--1` = один-к-одному, `*--*` = многие-ко-многим.

```
User (1) ──── (1) Profile
User (1) ──── (*) CreditApplication
User (*) ──── (*) Role            [через UserRole, Middle+]
Role (*) ──── (*) Permission      [через RolePermission, Middle+]

CreditProduct (1) ──── (*) CreditApplication

CreditApplication (1) ──── (0..1) ScoringResult
CreditApplication (1) ──── (*) Document          [Middle+]
CreditApplication (1) ──── (0..1) Loan           [создаётся при DISBURSED]

Loan (1) ──── (*) PaymentScheduleItem
Loan (1) ──── (*) Transaction

User (1) ──── (*) Notification    [Middle+]
User (1) ──── (*) AuditLog        [actor, Middle+]
```

**Матрица «уровень → какие сущности вводятся»:**

| Сущность | Junior | Middle | Senior |
|---|:---:|:---:|:---:|
| User | ✅ (создаётся) | ← | рефакторинг в Aggregate |
| Profile | ✅ | ← | ← |
| CreditProduct | ✅ | ← | ← |
| CreditApplication | ✅ | расширяется статусами | Aggregate Root |
| ScoringResult | ✅ (синхронный) | async (Celery) | FastAPI-сервис |
| Role / Permission | — | ✅ | ← |
| Loan | — | ✅ | Aggregate Root |
| PaymentScheduleItem | — | ✅ | ← |
| Transaction | — | ✅ | ← |
| Document | — | ✅ (local FS) | ✅ (S3) |
| Notification | — | ✅ | ← |
| AuditLog | — | ✅ | ← + immutable |
| FeatureFlag | — | — | ✅ |

> Ключевой принцип: **сущности Junior не переписываются**. Middle добавляет к ним внешние ключи и новые таблицы; Senior меняет способ организации кода (слои), но структура данных остаётся обратно совместимой через миграции.

### 3.3 Машина состояний кредитной заявки

```
        ┌────────┐  submit   ┌───────────┐  auto      ┌──────────┐
        │ DRAFT  │──────────▶│ SUBMITTED │───────────▶│ SCORING  │
        └────────┘           └───────────┘            └────┬─────┘
             ▲                                              │
             │ edit                        ┌────────────────┼────────────────┐
             │                             ▼                ▼                ▼
             │                       ┌──────────┐    ┌──────────────┐  ┌──────────┐
             └───────────────────────│ REJECTED │    │ MANUAL_REVIEW│  │ APPROVED │
                                     └──────────┘    └──────┬───────┘  └────┬─────┘
                                                            │               │
                                              approve/reject│               │ disburse
                                                            ▼               ▼
                                                     (APPROVED/REJECTED)  ┌──────────┐
                                                                          │ DISBURSED│
                                                                          └──────────┘
```

- **Junior:** реализует `DRAFT → SUBMITTED → SCORING → APPROVED/REJECTED` (скоринг синхронный, MANUAL_REVIEW не используется).
- **Middle:** добавляет `MANUAL_REVIEW`, `DISBURSED`, ручную обработку андеррайтером, асинхронный скоринг.
- **Senior:** переходы становятся доменными событиями (`ApplicationApproved`, `LoanDisbursed`), публикуемыми в брокер.

## 5. Стандарты API

### 5.1 Базовые соглашения

- **Base URL:** `/api/v1/`
- **Формат:** JSON (UTF-8).
- **Именование:** ресурсы — существительные во множественном числе (`/credit-applications`), kebab-case в URL, snake_case в JSON-полях.
- **Версионирование:** через префикс пути (`/api/v1/`). Ломающие изменения → `/api/v2/`.
- **Идемпотентность:** GET, PUT, DELETE идемпотентны; POST — нет.

### 5.2 Единый формат успешного ответа

Одиночный ресурс:
```json
{
  "id": 42,
  "amount": "200000.00",
  "status": "SUBMITTED",
  "created_at": "2026-07-09T10:22:00Z"
}
```

Коллекция (пагинация):
```json
{
  "count": 137,
  "next": "/api/v1/credit-applications/?page=3",
  "previous": "/api/v1/credit-applications/?page=1",
  "results": [ { "id": 42, "...": "..." } ]
}
```

### 5.3 Единый формат ошибки

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Amount exceeds product maximum",
    "details": {
      "amount": ["Must be less than or equal to 500000.00"]
    },
    "trace_id": "b3f1c2a4-..."
  }
}
```

Коды ошибок (сквозные): `VALIDATION_ERROR`, `AUTHENTICATION_FAILED`, `TOKEN_EXPIRED`, `PERMISSION_DENIED`, `NOT_FOUND`, `CONFLICT`, `RATE_LIMITED`, `INTERNAL_ERROR`.

### 5.4 Таблица HTTP-статусов

| Статус | Когда используется |
|---|---|
| 200 OK | Успешный GET / PUT / PATCH |
| 201 Created | Успешный POST (создание) |
| 204 No Content | Успешный DELETE |
| 400 Bad Request | Ошибка валидации |
| 401 Unauthorized | Нет/невалидный/истёкший токен |
| 403 Forbidden | Аутентифицирован, но нет прав |
| 404 Not Found | Ресурс не найден / не принадлежит пользователю |
| 409 Conflict | Конфликт состояния (напр. повторная отправка заявки) |
| 422 Unprocessable | Семантически неверные данные (бизнес-правило) |
| 429 Too Many Requests | Превышен rate limit (Senior) |
| 500 Internal Server Error | Непредвиденная ошибка сервера |

### 5.5 Pagination / Filtering / Sorting / Search (сквозные query-параметры)

| Параметр | Пример | Описание |
|---|---|---|
| `page` | `?page=2` | Номер страницы (page-based pagination) |
| `page_size` | `?page_size=20` | Размер страницы (default 20, max 100) |
| `ordering` | `?ordering=-created_at` | Сортировка (`-` = убывание) |
| `search` | `?search=потреб` | Полнотекстовый поиск по заданным полям |
| фильтры | `?status=SUBMITTED&min_amount=100000` | Фильтрация по полям |

## 6. Соглашения безопасности

### 6.1 JWT

- **Access token:** JWT, TTL 15 минут, передаётся в заголовке `Authorization: Bearer <token>`.
- **Refresh token:** JWT, TTL 7 дней. На Junior хранится клиентом; на Senior — rotation + blacklist.
- **Payload access-токена:**
```json
{
  "token_type": "access",
  "user_id": 42,
  "role": "CLIENT",
  "exp": 1751000000,
  "jti": "..."
}
```

### 6.2 Поток аутентификации (общий)

```
Регистрация:
  Client → POST /auth/register {email, password}
         ← 201 {user}
Логин:
  Client → POST /auth/login {email, password}
         ← 200 {access, refresh}
Доступ к защищённому ресурсу:
  Client → GET /credit-applications  [Authorization: Bearer access]
         ← 200 {results}
Истёк access:
  Client → GET ...  ← 401 TOKEN_EXPIRED
  Client → POST /auth/refresh {refresh}
         ← 200 {access}
  Client → повтор исходного запроса
Истёк refresh:
  Client → POST /auth/refresh  ← 401
  → полный logout, редирект на /login
```

## 7. Матрица соответствия Backend ↔ Frontend

Каждая страница фронтенда сопоставлена с backend-эндпоинтами. Полная детализация — в документах уровней; здесь сводная карта.

| Frontend-страница | Уровень | Backend API |
|---|---|---|
| `/register` | Junior | `POST /auth/register` |
| `/login` | Junior | `POST /auth/login`, `POST /auth/refresh` |
| `/profile` | Junior | `GET/PUT /profile`, `POST /profile/avatar` |
| `/products` | Junior | `GET /credit-products` |
| `/products/[id]` | Junior | `GET /credit-products/{id}` |
| `/applications` | Junior | `GET /credit-applications` |
| `/applications/new` | Junior | `POST /credit-applications` |
| `/applications/[id]` | Junior | `GET/PUT/DELETE /credit-applications/{id}`, `POST /{id}/submit` |
| `/loans` | Middle | `GET /loans` |
| `/loans/[id]` | Middle | `GET /loans/{id}`, `GET /loans/{id}/schedule`, `POST /loans/{id}/repay` |
| `/notifications` | Middle | `GET /notifications`, WS `/ws/notifications` |
| `/admin/products` | Middle | `GET/POST/PUT/DELETE /admin/credit-products` |
| `/admin/applications` | Middle | `GET /admin/applications`, `POST /{id}/approve`, `POST /{id}/reject` |
| `/admin/users` | Middle | `GET /admin/users`, `PATCH /admin/users/{id}/role` |
| `/admin/audit` | Middle | `GET /admin/audit-log` |

**Правило полноты:** к финальному Senior-уровню **каждый** backend-эндпоинт обязан использоваться хотя бы одной страницей фронтенда. Матрица покрытия проверяется в acceptance-критериях.
